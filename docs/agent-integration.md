# Agent Integration Guide

Zoro is designed from the ground up to be **agent-friendly**. Unlike traditional reference managers that lock data behind proprietary databases, Zoro stores papers in a human-readable, filesystem-based layout that AI agents can navigate, read, and extend without any special API.

## Why Zoro is Agent-Friendly

- **Filesystem-first storage**: Papers are stored as plain directories with JSON metadata files -- no proprietary formats
- **Predictable structure**: Every paper directory follows the same layout (`metadata.json`, `attachments/`, `notes/`)
- **Agent-writable directories**: The `attachments/` and `notes/` directories are designed for agents to write summaries, translations, and analysis
- **Stable slug identifiers**: Paper slugs are deterministic and human-readable, making them easy to reference in conversation
- **Standard formats**: Metadata is JSON, papers are PDF/HTML, notes are Markdown

## File Structure

All Zoro data lives under `~/.zoro/`:

```
~/.zoro/
  config.toml                              # App configuration
  library.db                               # SQLite database (source of truth)

  library/
    papers/
      2017-attention-is-all-you-need-a1b2c3d4/
        metadata.json                      # Paper metadata (structured JSON)
        paper.pdf                          # PDF version (if downloaded)
        abs.html                         # HTML version (ArXiv HTML, etc.)
        attachments/                       # Agent-writable directory
          summary.md                       #   AI-generated summary
          translation-zh.md                #   AI-generated translation
          key-findings.md                  #   Extracted key findings
        notes/                             # User and agent notes
          reading-notes.md                 #   User's reading notes
      2024-scaling-laws-for-neural-b5c6d7e8/
        metadata.json
        paper.pdf
        attachments/
        notes/
    collections/                           # Collection metadata

  subscriptions/
    cache/                                 # Subscription feed caches

  exports/                                 # BibTeX/RIS export output
  logs/                                    # Application logs
```

## Paper Slug Format

Each paper is identified by a **slug** -- a deterministic, human-readable string:

```
{year}-{title-slug}-{8char-sha256-hash}
```

Examples:
- `2017-attention-is-all-you-need-a1b2c3d4`
- `2024-scaling-laws-for-neural-language-b5c6d7e8`
- `2023-llama-open-and-efficient-foundation-9f8e7d6c`

The slug is generated by `generate_paper_slug()` in `crates/zoro-core/src/slug_utils.rs`:

```rust
pub fn generate_paper_slug(title: &str, identifier: &str, year: Option<&str>) -> String {
    let year_str = year
        .map(|y| y[..4].to_string())
        .unwrap_or_else(|| chrono::Utc::now().format("%Y").to_string());

    let title_slug = slugify(title);
    let truncated = truncate_on_word_boundary(&title_slug, 40);

    let mut hasher = Sha256::new();
    hasher.update(identifier.as_bytes());
    let hash = format!("{:x}", hasher.finalize());
    let short_hash = &hash[..8];

    format!("{}-{}-{}", year_str, truncated, short_hash)
}
```

- **Year**: Extracted from `published_date` or defaults to the current year
- **Title slug**: Slugified (lowercase, hyphenated), truncated to 40 characters on a word boundary
- **Hash**: First 8 characters of the SHA-256 hash of the paper's identifier (DOI, ArXiv ID, or title)

This format ensures slugs are:
- **Human-readable**: You can identify the paper by its slug
- **Deterministic**: The same paper always produces the same slug
- **Unique**: The hash suffix prevents collisions
- **Filesystem-safe**: Only lowercase letters, digits, and hyphens

## `metadata.json` Schema

Each paper directory contains a `metadata.json` file with the following structure (from `PaperMetadata` in `models.rs`):

```json
{
  "id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "slug": "2017-attention-is-all-you-need-a1b2c3d4",
  "title": "Attention Is All You Need",
  "authors": [
    {
      "name": "Ashish Vaswani",
      "affiliation": "Google Brain",
      "orcid": null
    },
    {
      "name": "Noam Shazeer",
      "affiliation": "Google Brain",
      "orcid": null
    }
  ],
  "abstract": "The dominant sequence transduction models are based on complex recurrent or convolutional neural networks...",
  "doi": "10.48550/arXiv.1706.03762",
  "arxiv_id": "1706.03762",
  "url": "https://arxiv.org/abs/1706.03762",
  "pdf_url": "https://arxiv.org/pdf/1706.03762",
  "html_url": "https://arxiv.org/html/1706.03762",
  "published_date": "2017-06-12T00:00:00Z",
  "added_date": "2024-01-15T10:30:00Z",
  "source": "browser-extension",
  "tags": ["transformer", "attention", "nlp"],
  "collections": ["Machine Learning"],
  "attachments": [
    {
      "filename": "paper.pdf",
      "type": "pdf",
      "created": "2024-01-15T10:30:00Z"
    }
  ],
  "notes": ["reading-notes.md"],
  "read_status": "read",
  "rating": 5,
  "extra": {
    "labels": ["cs.AI", "Machine Learning"]
  }
}
```

### Field Reference

| Field | Type | Description |
|---|---|---|
| `id` | `string` | UUID v4 identifier |
| `slug` | `string` | Filesystem-safe slug (directory name) |
| `title` | `string` | Paper title |
| `authors` | `Author[]` | List of authors with name, affiliation, ORCID |
| `abstract` | `string?` | Paper abstract (JSON key is `"abstract"`, not `"abstract_text"`) |
| `doi` | `string?` | Digital Object Identifier |
| `arxiv_id` | `string?` | ArXiv paper ID (e.g., `"1706.03762"`) |
| `url` | `string?` | Canonical URL |
| `pdf_url` | `string?` | Direct PDF download URL |
| `html_url` | `string?` | HTML version URL |
| `published_date` | `string?` | ISO 8601 publication date |
| `added_date` | `string` | RFC 3339 timestamp when added to library |
| `source` | `string?` | How the paper was added (`"browser-extension"`, `"subscription"`, `"manual"`, `"import"`) |
| `tags` | `string[]` | Tag names |
| `collections` | `string[]` | Collection names |
| `attachments` | `AttachmentInfo[]` | List of attached files |
| `notes` | `string[]` | List of note filenames |
| `read_status` | `string` | `"unread"`, `"reading"`, or `"read"` |
| `rating` | `number?` | 1-5 star rating |
| `extra` | `object` | Arbitrary JSON for extensibility. May contain `labels: string[]` for read-only metadata labels (arXiv categories, Zotero tags, etc.) |

## Attachments Directory

The `attachments/` subdirectory in each paper folder is designed for **agents to write files**. Common use cases:

| File | Purpose |
|---|---|
| `summary.md` | AI-generated paper summary |
| `translation-zh.md` | Chinese translation of the abstract/paper |
| `translation-ja.md` | Japanese translation |
| `key-findings.md` | Extracted key findings and contributions |
| `critique.md` | Critical analysis |
| `related-work.md` | Related paper connections |
| `code-review.md` | Analysis of associated code |

Agents can create any files they want in this directory. The naming convention is descriptive kebab-case with a `.md` extension for text content.

## Notes Directory

The `notes/` subdirectory stores user and agent notes as Markdown files. Unlike `attachments/` (which is for derived content), `notes/` is for commentary and annotations.

## Example Workflows

### Reading Paper Metadata with an AI Agent

An AI coding agent (like Claude Code or OpenClaw) can read paper metadata directly from the filesystem:

```bash
# List all papers in the library
ls ~/.zoro/library/papers/

# Read metadata for a specific paper
cat ~/.zoro/library/papers/2017-attention-is-all-you-need-a1b2c3d4/metadata.json

# Read the abstract using jq
cat ~/.zoro/library/papers/2017-attention-is-all-you-need-a1b2c3d4/metadata.json | jq '.abstract'

# Find all papers by a specific author
for dir in ~/.zoro/library/papers/*/; do
  if jq -e '.authors[] | select(.name | contains("Vaswani"))' "$dir/metadata.json" > /dev/null 2>&1; then
    echo "$dir"
  fi
done
```

### Automated Summarization Pipeline

An agent can generate summaries for papers that don't have them:

```bash
# Find papers without summaries
for dir in ~/.zoro/library/papers/*/; do
  if [ ! -f "$dir/attachments/summary.md" ]; then
    echo "Needs summary: $(jq -r '.title' "$dir/metadata.json")"
  fi
done
```

Then for each paper, the agent can:
1. Read `metadata.json` for title, abstract, and context
2. Read `abs.html` or `paper.pdf` for full text
3. Generate a summary
4. Write it to `attachments/summary.md`

### Batch Processing Papers

```python
import json
import os
from pathlib import Path

papers_dir = Path.home() / ".zoro" / "library" / "papers"

for paper_dir in papers_dir.iterdir():
    if not paper_dir.is_dir():
        continue
    
    metadata_path = paper_dir / "metadata.json"
    if not metadata_path.exists():
        continue
    
    with open(metadata_path) as f:
        metadata = json.load(f)
    
    print(f"Paper: {metadata['title']}")
    print(f"  Authors: {', '.join(a['name'] for a in metadata['authors'])}")
    print(f"  Status: {metadata['read_status']}")
    print(f"  Tags: {', '.join(metadata['tags'])}")
    
    # Check for existing attachments
    attachments_dir = paper_dir / "attachments"
    if attachments_dir.exists():
        for attachment in attachments_dir.iterdir():
            print(f"  Attachment: {attachment.name}")
```

### Using the SQLite Database Directly

For more complex queries, agents can query the SQLite database:

```bash
# Count papers by read status
sqlite3 ~/.zoro/library.db "SELECT read_status, COUNT(*) FROM papers GROUP BY read_status"

# Full-text search
sqlite3 ~/.zoro/library.db "SELECT title FROM papers WHERE rowid IN (SELECT rowid FROM papers_fts WHERE papers_fts MATCH 'transformer attention')"

# Recent papers
sqlite3 ~/.zoro/library.db "SELECT title, added_date FROM papers ORDER BY added_date DESC LIMIT 10"
```

## Best Practices for Agent Integration

1. **Read `metadata.json`, not the database** for individual paper metadata. The JSON file is the stable, agent-oriented interface.

2. **Use the database for queries**. When you need to search, filter, or aggregate across papers, the SQLite database with FTS5 is more efficient than scanning the filesystem.

3. **Write to `attachments/`**, not to the paper directory root. The root contains Zoro-managed files (`metadata.json`, `paper.pdf`, `abs.html`).

4. **Use descriptive filenames** for attachments. Prefer `summary.md` over `output.txt`, and `translation-zh.md` over `translated.md`.

5. **Do not modify `metadata.json` directly**. The database is the source of truth. Changes to `metadata.json` will be overwritten when the app syncs. Use the Tauri commands or connector API to update metadata.

6. **Use slugs for stable references**. Paper slugs are deterministic and don't change after creation. Use them in conversations and references.

7. **Check for existing content before generating**. Before generating a summary, check if `attachments/summary.md` already exists.

8. **Use standard Markdown** for generated content. Zoro may render Markdown in the UI in the future.
